# V1 Build Plan — Step by Step

> The complete stage-by-stage specification for V1. Each stage lists exactly what it does, what model runs it, what it produces, and any constraints. Build this.

---

## Stage 1 — Intake & Decomposition

**Model:** Gemini 3.1 Pro
**Fallback:** GPT-5.4 mini or Claude Sonnet 4.6 (Gemini is still in preview — if unstable, swap at config level)

**What it does:** Takes the user's raw question and breaks it into structured components the rest of the pipeline operates on.

**Steps:**
1. Restate the query — show the restated question in terminal so the user can re-run if it's wrong
2. Break into empirical sub-questions (what can be researched)
3. Map optimization axes / tradeoffs (feeds Stage 6 tradeoff output)
4. Generate research plan: what to verify, what sources matter, what evidence would change the recommendation
5. Generate falsification targets: what would prove the recommendation wrong (N/A is valid for non-falsifiable topics)
6. Prompt self-check: "Does this decomposition address the FULL question? Are sub-questions neutral, not leading?"
7. Emit stage summary with `reasoning` field

**Output artifacts:**
- Decomposition object (sub-questions, optimization axes, research plan, falsification targets)
- Stage summary (goal, key findings, decisions, outcome, reasoning)

**Note:** No decomposition validation stage in V1. The self-check prompt instruction and user terminal review are the mitigations.

## Stage 2 — Broad Retrieval & Evidence Normalization

**Tools:** Tavily + Exa (routed by query type — see Step 3 below)

**What it does:** For each sub-question from Stage 1, searches the web for real evidence and normalizes it into a structured evidence package.

**Steps:**

**Query generation:**
1. Generate 3-4 query variants per sub-question via lightweight model call (same model used for finding extraction — GPT-5.4 mini or Claude Haiku 4.5). The model generates variants using complementary diversification strategies — each strategy retrieves different documents, so combining them yields broader coverage than any single approach:
   - **Keyword rewrite:** Extract search-engine-effective keywords. Strip filler, keep specifics.
   - **Practitioner rewrite:** Rewrite as a practitioner would search — include signals like "lessons learned", "we found that", specific tool/library names, or `site:github.com`.
   - **Contrarian/falsification:** Generate a query that would find evidence AGAINST the expected answer — limitations, alternatives, failures.
   - **Semantic description (Exa only):** A 1-2 sentence description of the ideal document, as if describing a link you want to find. Leverages Exa's next-link prediction architecture.
   See `V1_PROMPTS.md` Stage 2 query generation prompt for the full template.

**API-specific formatting:**
2. Format queries for target API before execution:
   - **Tavily queries:** Keyword-oriented, under 400 characters. Do not use `site:` operators in query text — use `include_domains` API parameter instead.
   - **Exa queries:** Frame as descriptions of the content you want to find, not keyword strings. Example: "Here is the official API documentation for configuring Tavily search parameters:" outperforms "Tavily API docs."
   - **Exa `systemPrompt`:** Set to "Prefer official sources, primary documentation, and peer-reviewed research over blog posts, aggregator content, and secondary summaries." (This is a first-class Exa API parameter for `type: "deep"` searches.)

**Domain filtering (API-level):**
3. Apply domain filtering via API parameters (not in query text):
   - Default `exclude_domains` on all queries: `["medium.com", "towardsdatascience.com", "dev.to"]` — aggregator/secondary sources.
   - Optional topic-specific `include_domains` (≤5 domains per Tavily guidance): e.g., for ML research `["arxiv.org", "openreview.net"]`, for cloud APIs `["docs.aws.amazon.com", "cloud.google.com/docs"]`.
   - Tavily: `include_domains`/`exclude_domains` parameters. Exa: `includeDomains`/`excludeDomains` parameters.

**Dual-API routing:**
4. Route queries to Tavily or Exa based on query type:

   | Query type | API | Key parameters |
   |-----------|-----|----------------|
   | Default/keyword | Tavily | `search_depth="basic"`, filter results by `score > 0.7` |
   | News/current events | Tavily | `topic="news"`, `time_range="week"` (adds `published_date` metadata) |
   | Semantic/conceptual | Exa | `type="deep"`, descriptive query, `systemPrompt` set |
   | Academic | Exa | `category="research paper"` |
   | Content extraction (known URLs) | Tavily Extract | 1 credit per 5 URLs |
   | High-priority sub-questions | Both in parallel | `asyncio.gather()`, deduplicate results by normalized URL |

   **Note (January 2026):** Tavily also supports `search_depth="fast"` (lower latency than basic, supports `chunks_per_source`) and `search_depth="ultra-fast"` (minimum latency, single NLP summary per URL). V1 uses `basic` and `advanced` only. Consider `fast` as a potential upgrade if Stage 2 latency is a bottleneck — it offers chunk support at lower cost than `advanced`.

   Cost controls:
   - Tavily: Always set `search_depth` explicitly — `auto_parameters` can silently upgrade to `advanced` (2x cost). Use `basic` by default (1 credit).
   - Exa: Contents are now returned by default with search results (no separate contents call needed). Use `highlights` with `max_characters` parameter to control highlight length (legacy `numSentences` and `highlightsPerUrl` are deprecated). Note: The Exa SDK surface changed in early 2026 — verify exact parameter names in current exa-py docs before implementation. Set `maxAgeHours=-1` for static/historical content to avoid livecrawl costs.
   - Exa pricing (as of March 2026): standard search $0.007/request (contents included for 10 results), `type="deep"` $0.012/request, `type="deep-reasoning"` $0.015/request. Using `type="deep"` for semantic queries (per the routing table) costs ~70% more than `type="auto"` — worth it for systemPrompt control. Reserve deep search for sub-questions where domain preference matters.
   - Batch concurrent queries with `asyncio.gather()`.

**Search execution:**
5. Execute searches — hard cap: 4 queries per sub-question (named constant `MAX_QUERIES_PER_SUB_QUESTION`)

**URL deduplication (before quality scoring):**
6. Normalize all result URLs: strip query parameters (except meaningful ones like `?page=` or `?id=`), remove trailing slashes, remove `www.` prefix, lowercase the domain. Deduplicate by normalized URL — keep the occurrence with the highest API relevance score. Implementation: `urllib.parse` + custom param stripping (~10 lines). Prevents the same page from consuming multiple budget slots.

**Source quality scoring:**
7. Apply source quality scoring via URL lookup table (authority score). Unknown sources default to 0.5 (not penalized). Note: Tavily's `score` field is relevance, not credibility — do not use as quality score.

   | Source type | Authority score | Targeting examples |
   |---|---|---|
   | Official docs | 1.0 | Domains with `/docs/`, `/documentation/`, `/api/` paths |
   | Academic papers | 0.9 | `arxiv.org`, `aclanthology.org`, `.edu`, DOI presence |
   | GitHub repos/issues | 0.8 | `github.com` with discussion/issue context |
   | Practitioner posts with data | 0.7 | Named author + code/benchmarks + dates |
   | Forum discussion (HN, Reddit) | 0.5 | `news.ycombinator.com`, `reddit.com` |
   | Generic blog/news | 0.3 | Last resort |

**Freshness-adjusted scoring:**
8. Blend the authority score with a freshness signal. The Source schema's `quality_score` field stores the FINAL blended result, not just the lookup table value.

   ```
   HALF_LIFE_DAYS = {
       "ai_models": 365,      # ~1 year
       "software": 730,       # ~2 years
       "medicine": 1825,      # ~5 years
       "economics": 3285,     # ~9 years
       "physics": 4745,       # ~13 years
       "default": 1095        # ~3 years
   }

   freshness = 0.5 ** (age_days / half_life_days)

   TEMPORAL_WEIGHT = {
       "ai_models": 0.5,      # freshness matters a lot
       "software": 0.4,
       "default": 0.2,        # most topics: authority matters more
       "physics": 0.05        # freshness barely matters
   }

   quality_score = (temporal_weight * freshness) + ((1 - temporal_weight) * authority_score)
   ```

   **Authority floor:** `quality_score = max(quality_score, 0.4)` for sources matching authority floor patterns (RFCs, W3C specs, seminal papers, official specifications). These never drop below 0.4 regardless of age.

   The topic→half_life mapping is a simple dict lookup. Start with `"default"` for V1 and tune per-topic if needed. Requires `publication_date` on Source (from Exa `publishedDate`, Tavily `published_date` with `topic="news"`, or extracted from content). If no publication date is available, skip freshness adjustment and use authority score directly.

**Staleness flags:**
9. Apply three regex checks to the first 2000 characters of each source's content. These are scoring modifiers applied during `quality_score` computation, not separate schema fields:
   - **Deprecation keywords:** Scan for "deprecated", "end-of-life", "no longer maintained", "use [X] instead", "this page has moved". If found, reduce `quality_score` by 0.3 (floor at 0.1).
   - **Version-in-URL:** Extract version numbers from URL path with `r'/v(\d+)/'` pattern. Flag if version < known current version (requires optional config dict of `{api_name: current_version}` — skip if not configured).
   - **Year-mention:** Extract all 4-digit years from content. If max year mentioned is >2 years before current year AND topic half-life < 3 years, reduce `quality_score` by 0.15.

**Finding extraction:**
10. Extract atomic findings from each source — this is quality bottleneck #1, no shortcuts. If Tavily/Exa content needs interpretation to produce atomic findings, use a lightweight model call (see `V1_PROMPTS.md` Stage 2 finding extraction prompt). If API output is already structured with discrete facts, orchestrator can extract programmatically.

**Compression and sufficiency:**
11. Basic compression: keep findings and preserve key quotes
12. Check evidence sufficiency: minimum 2 independent sources per sub-question. This check runs AFTER URL dedup (Step 6) — deduped URLs don't count as independent sources.
13. Emit stage summary with `reasoning` field

**Practitioner-signal patterns:** `"we found that"`, `"lessons learned"`, `site:github.com`
**Avoid:** "best practices for X," "ultimate guide to Y"

**Output artifacts:**
- Evidence package (normalized sources with blended quality scores, per sub-question)
- Stage summary (goal, sources found, quality assessment, gaps identified, reasoning)

## Stage 3 — Independent Candidate Generation

**Models (parallel, truly independent):**
- GPT-5.4 — landscape frame (step-back abstraction)
- Claude Opus 4.6 — critical frame (verification-first)
- Gemini 3.1 Pro — verification frame (structured decomposition)

**Fallback:** If Gemini is unstable, use GPT-5.4 mini or Claude Sonnet 4.6. Stage requires ≥2 of 3 models to succeed.

**What it does:** Each model independently analyzes the evidence package using a different reasoning frame. No model sees another's output.

**Each model must produce:**
1. Bottom-line recommendation (prevents hedging)
2. Specific, evidence-linked claims
3. Explicit assumptions
4. Evidence references (linked to Stage 2 sources)
5. Counter-argument against their own recommendation (required non-empty string — prompt asks for specifics but no automated validation)
6. What would falsify their recommendation
7. Confidence level (HIGH/MEDIUM/LOW — telemetry only, not arbitration input)
8. `reasoning` field

**Anonymization:** Outputs are labeled A/B/C via string replacement. No model identity leaks downstream.
**Anti-conformity:** Models may only change position when citing new evidence, a corrected assumption, or a resolved contradiction. Prompt never asks "who is more convincing."

**Output artifacts:**
- Three independent analysis objects (structured JSON, anonymized A/B/C)
- Stage summary per model (goal, key claims, reasoning approach, confidence level, reasoning)

## Stage 4 — Claim Extraction & Dispute Localization

**Model:** GPT-5.4 (best structured output via constrained decoding)
**Fallback:** Claude Opus 4.6

**What it does:** Decomposes the three independent analyses into atomic claims, identifies where they conflict, classifies each conflict by type, and filters for decision-criticality. This is quality bottleneck #2.

**Steps:**

**4a. Claim extraction:**
1. Decompose each model's analysis into atomic claims using Claimify approach (sentence tokenization → context creation → LLM extraction → disambiguation)
2. Extract assumptions as first-class claims
3. Deduplicate claims across the three analyses (LLM-based, not programmatic)
4. Assign global IDs and evidence labels to all claims (evidence label hierarchy: vendor-documented > empirically observed > model self-characterization > speculative inference)

**Primacy bias mitigation (before sending to model):**
- Randomize the presentation order of Analyst A/B/C before each Stage 4 call. The first analysis in the prompt gets disproportionate weight during extraction and deduplication. Shuffle order per call so no model is systematically advantaged.
- For Stage 5 arbitration calls: also randomize the order of analyst positions per dispute.
- Preferred approach: if token budget allows, extract claims from each analysis in a separate call (one model at a time), then cross-reference in a second pass. This eliminates primacy bias entirely by removing the comparison context during extraction. The prompt already notes this option ("the orchestrator may split this into multiple calls").

**4b. Dispute localization:**
5. Identify where the three analyses conflict
6. Classify each conflict:
   - **Empirical** — dispute about facts → resolve with more evidence (Stage 5)
   - **Interpretive** — dispute about what evidence means → arbitration critique (Stage 5)
   - **Preference-weighted** — dispute about optimization criteria → user context needed (Stage 6a)
   - **Spec ambiguity** — question not constrained enough → user steering (Stage 6a)
   - **Other** — edge case not fitting above types, with free-text description field

**4c. Decision-criticality gate:**
7. Filter: does this disagreement affect which alternative is recommended? If not, log but don't route to resolution.

**4d. Resolution routing:**
8. Route disputes deterministically based on type (empirical/interpretive → Stage 5, preference/ambiguity/other → Stage 6a)

9. Emit stage summary with `reasoning` field

**Output artifacts (all materialized, not hidden in prompt state):**
- Claim ledger: all extracted claims with source model (anonymized), evidence links, status
- Assumption set: all extracted assumptions, which claims depend on them
- Dispute queue: classified disagreements with type, criticality flag, resolution routing
- Stage summary (claims extracted count, disputes found, type breakdown, what got filtered, reasoning)

## Stage 5 — Targeted Verification & Arbitration

**Tools:** Tavily/Exa for evidence search
**Model for arbitration:** Claude Opus 4.6 (single-turn, schema-driven critique)

**What it does:** Only fires on unresolved disputes from Stage 4 that are both empirical/interpretive AND decision-critical. Low-conflict queries skip this stage entirely.

**Steps:**
1. For each routed dispute, generate neutral verification queries — queries that find the definitive answer without presupposing either position. Three query types per dispute:
   - **Neutral question:** Convert the dispute into a question that doesn't favor either side (AVeriTeC approach). Example: instead of "Einstein born 1879" or "Einstein born 1880", search "What year was Einstein born?"
   - **Supporting search for weaker position:** A query that would find evidence FOR the currently less-supported position. Counteracts confirmation bias — the pipeline naturally finds evidence for the leading claim more easily.
   - **Authoritative source:** A query targeting the most authoritative source type for this dispute — official docs, specs, peer-reviewed research. Use `include_domains` if a specific authoritative domain is known.
   See `V1_PROMPTS.md` Stage 5 query generation template for the full specification.
2. Search for both supporting AND disconfirming evidence. Use Tavily `search_depth="advanced"` with `chunks_per_source=3` for targeted content chunks.
3. Arbitration: update claim status based on new evidence (verified / refuted / unresolved). Arbitration prompt includes explicit debiasing instruction — see `V1_PROMPTS.md` Stage 5 SYSTEM prompt.
4. Update claim ledger and dispute queue with resolution status
5. Emit stage summary with `reasoning` field

**Budget:** Max 3 queries per disputed claim. Max 2 rounds total.

**Recency-sensitive claims:** Require official source OR 2 independent timestamped sources.

**Output artifacts:**
- Updated claim ledger + dispute queue with resolution status
- Stage summary (disputes investigated, evidence found, what changed, what remains unresolved, reasoning)

### Reference Implementations (study material for developer)

**For Stage 4 (claim extraction):** The Claimify approach used in Stage 4 has two small open-source implementations that can be studied for implementation patterns:
- `deshwalmahesh/claimify` (GitHub) — Python library, provider-agnostic. 3-stage pipeline (selection → disambiguation → decomposition). Most relevant as a code reference for the extraction logic.
- `AdamGustavsson/ClaimsMCP` (GitHub) — MCP server, OpenAI-locked. 4-stage pipeline. Apache 2.0 license.
Neither is mature enough to adopt as a dependency (7-13 stars, single contributors). Use as implementation references, not libraries.

**For Stage 5 (targeted verification):** Google SAFE (`google-deepmind/long-form-factuality`, ~677 stars, Apache 2.0, NeurIPS 2024) implements a search-augmented verification loop (generate query → retrieve → evaluate support) that closely mirrors Stage 5's pattern. SAFE uses SerperAPI for search; V1 uses Tavily/Exa — the swap is localized to one file. Study SAFE's loop architecture and edge case handling before building Stage 5 from scratch.

**For Stage 5 cost optimization (v1.1):** FIRE (`mbzuai-nlp/fire`, NAACL 2025) demonstrates dynamic retrieval depth — check model confidence before searching, and reuse knowledge from previous searches within the same query. V1's hard budget caps (3 queries/dispute, 2 rounds) are simpler and appropriate for V1. Consider FIRE's approach as a v1.1 optimization if Stage 5 costs are high.

## Stage 6a — User-Steering Interrupt

**What it does:** Before synthesis, checks the dispute queue for unresolved spec ambiguity, preference-weighted, or other-typed conflicts. If found, asks the user for input. If none found, skips straight to Stage 6b.

**Steps:**
1. Python filter on dispute queue: any unresolved preference-weighted, spec-ambiguity, or other disputes?
2. If yes: print dispute info to terminal (what's ambiguous, the options, why it matters)
3. User provides free-text input or presses Enter for default
4. Max 2 clarification questions per query

**No model call.** No schemas. Just terminal I/O using data already in the dispute objects.

## Stage 6b — Synthesis & Final Report

**Model:** TBD (validate Claude Opus 4.6 vs GPT-5.4 in testing — must not be the model that dominated earlier stages)
**Fallback:** If primary fails, try alternate model.

**What it does:** Compresses prior-stage artifacts, synthesizes the final report adapting to dispute types, and runs output validation.

**Steps:**

**Context compaction:**
1. Compress prior-stage artifacts when input exceeds ~80K characters (~20K tokens). Original query at start, key disputes and evidence at end. See `V1_PROMPTS.md` Context Compaction section for the full priority order (what to keep vs. truncate first).

**Synthesis (adapts based on dispute resolution):**
2. Empirical disputes resolved → converge on best-supported recommendation
3. Interpretive disputes → present strongest interpretation, note alternatives
4. Preference-weighted → surface 2-3 viable paths with explicit "if X then Y" conditions
5. Remaining ambiguity → flag honestly as unresolved
6. Always: lead with strongest recommendation, present alternatives as secondary

**Output validation (2 automated checks):**
7. Grounding check (regex): does the recommendation cite claims from the claim ledger?
8. Zombie check: are any alternatives that were eliminated by critique still included?

9. Emit stage summary and `reasoning` field

**Output: Final report (11 components, 3 tiers) + trace file**

### Tier A: Executive (what the user reads first)
1. **Executive recommendation** (1-3 paragraphs) — bottom-line answer as a recommendation, not a survey
2. **Conditions of validity** — what would flip the recommendation
3. **Decision-relevant tradeoffs** — "If you optimize for X → A. If you optimize for Y → B."

### Tier B: Analytical (what the user reads for understanding)
4. **Disagreement map** — types of disagreements found, action taken on each
5. **Preserved alternatives** — only alternatives that survived critique, with conditions for preference
6. **Key assumptions** — for each: if wrong, what changes?
7. **Confidence assessment** — per major claim: HIGH / MEDIUM / LOW
8. **Process summary** — per-stage bullets (4-6 each) compiled from stage summaries. Scannable in 30 seconds.

### Tier C: Evidentiary (what the user reads for verification)
9. **Claim ledger excerpt** — decision-critical claims with final status and resolution path
10. **Evidence trail** — sources, quality scores, conflicts and how resolved
11. **Evidence gaps** — what the system looked for but couldn't find

### Trace File (separate from report)
Full log of every stage's artifacts, raw claim ledger, dispute queue, evidence sources, resolution paths. For debugging and deep inspection.

## Cross-Cutting Requirements

### Reasoning field
All model-output schemas must include a `reasoning` field. Applies to: StageSummary, AnalysisObject, SynthesisReport, ArbitrationAssessment.

### Fallback logic
Three must-succeed stages:
- Stage 1: Gemini → GPT-5.4 mini
- Stage 4: GPT-5.4 → Claude Opus 4.6
- Stage 6b: primary → alternate

Everything else skips gracefully. Stage 3 requires ≥2 of 3 models to succeed.

### Partial trace on abort
If the pipeline fails at any stage, emit a trace file with everything completed so far.

### Anonymization
String replacement of model identifiers before any cross-model comparison. No model knows which model produced which output.

### Anti-conformity
A model may only change position when citing new evidence, a corrected assumption, or a resolved contradiction. Critique prompts ask for flaws and missing evidence, never "who is more convincing."

## Known Limitations

These are accepted tradeoffs for a personal CLI tool:

- **No decomposition validation** — biased framing in Stage 1 passes through unchecked (mitigated by prompt self-check + user terminal review)
- **No echo detection** — multiple sources citing the same upstream study look like independent sources
- **Lighter output validation** — 2 of 7 anti-patterns checked at runtime (grounding + zombie). The other 5 are evaluation criteria, not runtime checks.
- **Counter-argument quality is prompt-only** — no automated check that counter-arguments are specific and evidence-linked
- **No self-preference bias guard** — synthesis model may favor analysis from same architecture
- **Approximate context sizing** — char count heuristic instead of token counting
- **Compounding risk** — no decomposition validation + no echo detection means biased framing AND inflated evidence can both pass through unchecked
- **Error amplification in parallel architecture** — research shows independent multi-agent systems can amplify errors up to 17.2x without structured cross-referencing (Google Research, 2025). Stages 4-6 are the error-correction mechanism. If Stage 4 extraction quality is poor, the pipeline can produce worse results than a single model. This is the architectural reason Stage 4 is quality bottleneck #2.
