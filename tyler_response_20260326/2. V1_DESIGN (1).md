# Evidence-Grounded Reasoning Orchestrator — V1 Design

> V1 architecture spec. Reflects pruning scorecard decisions (52 tasks, 3 reviewers).
> Full unpruned design preserved in `V0/` for reference. Build plan: `V1/1. V1_Build_Plan_Step_By_Step.md`.

## What This Is

An evidence-grounded reasoning orchestrator for questions where the information landscape is noisy, fast-moving, and full of shallow advice. It replaces the human as the cross-model synthesis layer — decomposing questions, gathering real signal, forcing genuine cross-model critique, classifying disagreements by type, and converging on the strongest current answer — while preserving the human's judgment only for genuine ambiguity the system cannot resolve.

### Why It Exists

For cutting-edge or nuanced topics, single-pass AI research fails. It parrots consensus, misses practitioner reality, and can't distinguish signal from hype. But the manual alternative — pasting between models, framing critique prompts, evaluating corrections, synthesizing positions — is slow, cognitively exhausting, and doesn't scale.

This tool automates the orchestration overhead so the user's judgment operates on better-prepared material, faster.

### Scope

- Personal use tool, not a product.
- Must work for fast-moving technical topics AND subjective/nuance-heavy questions.
- Designed to be handed to a developer for implementation.

### Non-Goals (architecture constraints, not just positioning)

- **Not a generic multi-agent debate tool.** Debate is selective, not the core mechanism.
- **Not a research aggregator.** It doesn't just collect sources — it arbitrates claims.
- **Not a lens/persona engine.** It doesn't preserve perspectives for balance — it converges on the strongest answer.
- **Not a wrapper that calls multiple models and merges.** The value is in the structured critique, claim extraction, and dispute routing, not in parallel generation alone.
- **Not a conversational argument simulator.** Models don't "talk to each other." Code orchestrates discrete stages.

## Operating Theory

A single AI model answering a research question in a noisy, fast-moving space will produce polished consensus that may be wrong. It has no way to check its own blind spots, distinguish hype from signal, or verify against fresh evidence.

This tool is an **evidence-grounded reasoning orchestrator**. It replaces the human as the cross-model synthesis layer — carrying context between models, forcing genuine critique, classifying disagreements, and converging on the strongest answer — while preserving the human's judgment only for genuine ambiguity that the system cannot resolve.

It uses the genuine capability differences between frontier models (different training data, different reasoning styles, different failure modes) combined with structured web research and selective critique.

**Key constraints from research + live exercise:**
- Debate doesn't universally improve results. It helps for verification and judgment, not for solution-finding. Used selectively.
- Source quality is one bottleneck. Claim-level arbitration is the other. Both must be designed.
- The critique-and-integrate step produces the most quality improvement — not initial generation.
- Disagreements are classified by type (empirical/interpretive/preference/ambiguity), not questions by type.
- **Baseline discipline.** The full pipeline must demonstrably outperform: (a) single best model, (b) best-of-N independent generation, and (c) independent-parallel + majority aggregation. The martingale proof (Choi et al., NeurIPS 2025) shows majority voting accounts for most gains attributed to multi-agent debate. If the pipeline doesn't beat these baselines, it's wasting compute.

## Design Constraints

*10 constraints validated by cross-model meta-analysis, research, and lived experience:*

1. **Independence first.** Independent analysis must be truly blind — no model sees another's output. Post-critique agreement is weaker signal than independent convergence. The system tracks and distinguishes these.
2. **Critique targets claims, not responses.** Schema-driven critique ("identify weaknesses in this claim set"), not conversational ("what did the other model get wrong?").
3. **Anonymize in both directions.** Models don't know which model produced which output during critique. Prevents sycophancy toward prestigious models and competitive posturing.
4. **Track lineage of thought changes.** For every claim: initial position → critique received → what changed → why. Required for synthesis quality and user trust.
5. **Evidence labeling hierarchy.** All claims labeled: vendor-documented > empirically observed > model self-characterization > speculative inference. No unlabeled claims. See SCHEMAS.md `EvidenceLabel` enum for canonical definitions and numeric weights.
6. **Novelty/delta detection.** Flag when a critique round produces no new claims, no resolved disputes, no testable consequences. This is the stopping signal. *(V1: simplified to hard-cap counters, not semantic detection.)*
7. **Avoid social framing.** Models are analyzers/critics/verifiers, not teammates or rivals.
8. **Separate working notes from locked decisions.** Don't prematurely canonize evolving conclusions.
9. **Anti-conformity as protocol rule.** A model may only change position when citing new evidence, a corrected assumption, or a resolved contradiction. Critique asks for flaws and missing evidence, never "who is more convincing."
10. **Context rot mitigation.** Progressive summarization between stages, original query at start/end of prompts. *(V1: char count heuristic ~80K chars ≈ ~20K tokens, not precise token budgets.)*

*All 10 constraints are respected in V1. Two are partially implemented: #6 (counters, not semantic detection) and #10 (char heuristic, not token budgets). All others are fully implemented.*

## Pipeline: 6 Stages

*Simplified from the 8-stage research design. Principle: simplify the runtime, not the state model. Each stage produces distinct, inspectable artifacts.*

**Observability requirement:** Every stage must emit a **stage summary** alongside its primary artifact. The summary is a structured object with: stage name, goal (what this stage was trying to do), key findings (what conflicts/tradeoffs/signals it found), decisions made (what it chose and why), and outcome (what it produced). These summaries serve two purposes:
- **In the final report:** Compiled into a process summary (Tier B, component 8) — scannable bullets showing how the tool arrived at its recommendation.
- **In the trace file:** Full stage artifacts + summaries concatenated into a detailed reasoning log for debugging and provenance tracing.

### Stage 1 — Intake & Decomposition

**Model:** Gemini 3.1 Pro (cost-efficient routing task)

Break the query into structured components:
- Core question or decision
- Empirical sub-questions (what can be researched)
- Likely optimization axes / tradeoffs
- Research plan: what to verify, what sources matter, what evidence would change the recommendation
- Falsification targets: what evidence would prove the recommendation wrong (N/A valid for non-falsifiable topics)

**Self-checking (compensates for no Stage 1v):** Before outputting, the prompt self-verifies: Does this decomposition address the FULL question? Are sub-questions neutral, not leading? The user also sees the restated question in the terminal and can re-run if wrong.

**Output artifacts:**
- Decomposition object (sub-questions, research plan, falsification targets)
- Stage summary (goal, key findings, decisions, outcome)

*Note: Decomposition validation (Stage 1v) is deferred to v2. V1 mitigations: prompt self-checking + user terminal review of restated question. See V1 Limitations section.*

### Stage 2 — Broad Retrieval & Evidence Normalization

**Tools:** Tavily + Exa (routed by query type — see Build Plan Stage 2 for routing table).

For each sub-question from Stage 1:
- Generate 3-4 query variants via lightweight model call using complementary diversification strategies (keyword, practitioner, contrarian, + Exa semantic)
- Route queries to Tavily (keyword/news) or Exa (semantic/academic) based on query type
- Apply domain filtering via API parameters (`exclude_domains` for aggregators, optional `include_domains` for topic-specific sources)
- URL-normalize and deduplicate results before scoring
- Source quality scoring: URL lookup table (authority score) blended with freshness decay and staleness flags → final `quality_score`. Unknown sources default to 0.5 (not penalized). Note: Tavily's `score` field is relevance, not credibility. LLM-based scoring deferred to v2.
- Extract atomic findings from each source
- Basic compression: keep findings and preserve key quotes

**THIS IS QUALITY BOTTLENECK #1.** Shallow sources here = polished noise downstream.

**Output artifacts:**
- Evidence package (normalized sources with blended quality scores, per sub-question)
- Stage summary (goal, sources found, quality assessment, gaps identified)

**Budget:** Hard cap: 4 queries per sub-question (named constant).
**Evidence sufficiency:** Minimum 2 independent sources per sub-question (checked after URL dedup).

### Stage 3 — Independent Candidate Generation

**Models:** GPT-5.4, Claude Opus 4.6, Gemini 3.1 Pro (parallel, truly independent)

Each model independently analyzes the evidence package using a **different reasoning frame** (step-back abstraction, structured decomposition, verification-first — see PROMPTS.md for assignments). No model sees another's output.

Each must produce structured output exposing:
- Claims (specific, falsifiable)
- Assumptions (stated explicitly)
- Evidence used (with source references from Stage 2)
- Forced counter-argument against their own recommendation (required non-empty string; prompt asks for specifics but no automated validation in V1 — downgraded from Tier 4 to Tier 5)
- What would falsify their recommendation
- Confidence level (HIGH / MEDIUM / LOW)
- Reasoning: how the model got from evidence to conclusion

**Output artifacts:**
- Three independent analysis objects (structured JSON per model, anonymized as A/B/C)
- Stage summary per model (goal, key claims made, reasoning approach used, confidence level, reasoning)

### Stage 4 — Claim Extraction & Dispute Localization

**Model:** GPT-5.4 (best structured output reliability)

This is the critical arbitration stage. It operates on the three independent analyses and produces the state the rest of the pipeline depends on.

**4a. Claim extraction:** Decompose each model's analysis into atomic claims using the Claimify approach (sentence tokenization → context creation → LLM extraction → disambiguation). Extract assumptions as first-class claims.

**4b. Dispute localization:** Identify where the three analyses conflict. For each conflict, classify:
- **Empirical:** Dispute about facts → resolve with more evidence
- **Interpretive:** Dispute about what evidence means → reasoning critique
- **Preference-weighted:** Dispute about optimization criteria → user context needed
- **Spec ambiguity:** Question not constrained enough → structured user input needed
- **Other:** Edge case not fitting the above types → uses existing description field for context. Routes to user input (Stage 6a) — if the system can't classify it, the user is the best judge.

**4c. Decision-criticality gate:** Filter: does this disagreement affect which alternative is recommended? If not, log but don't route to resolution.

**THIS IS QUALITY BOTTLENECK #2.** Good sources + bad extraction = polished noise.

**Output artifacts (all must be materialized, not hidden in prompt state):**
- **Claim ledger:** All extracted claims with source model (anonymized), evidence links, status
- **Assumption set:** All extracted assumptions, which claims depend on them
- **Dispute queue:** Classified disagreements with type, criticality flag, and resolution routing

These artifacts are the canonical state for everything downstream. Synthesis operates on these, not on raw model outputs.

- Stage summary (claims extracted count, disputes found, types breakdown, what got filtered by criticality gate)

### Stage 5 — Targeted Verification & Arbitration

**Tools:** Tavily/Exa for evidence search.
**Model for arbitration reasoning:** Claude Opus 4.6 (single-turn, schema-driven critique)

**Only fires on unresolved disputes from Stage 4 that are both empirical/interpretive AND decision-critical.** Low-conflict queries skip this stage.

For each routed dispute:
- Generate neutral verification queries that find the definitive answer without presupposing either position (AVeriTeC-style neutral framing), plus a supporting search for the weaker position to counteract confirmation bias
- Search for both supporting AND disconfirming evidence (Tavily `search_depth="advanced"` with `chunks_per_source=3`)
- Arbitration: update claim status based on new evidence (verified / refuted / unresolved). Includes explicit debiasing instruction in the arbitration prompt.

**Budget:** Max 3 queries per disputed claim, max 2 rounds total.

**Output artifacts:**
- Updated claim ledger + dispute queue with resolution status
- Stage summary (disputes investigated, evidence found, what changed, what remains unresolved)

### Stage 6 — Synthesis & Report

**Model:** TBD (validate Claude Opus 4.6 vs GPT-5.4 in testing — must not be the model that dominated earlier stages)

**Conditional user-steering interrupt:** Before synthesis commits, check the dispute queue for unresolved spec ambiguity, preference-weighted, or other-typed conflicts. If found:
- Print dispute info to terminal: what's ambiguous, what the options are, why it matters
- User provides free-text input or presses Enter for default
- Max 2 clarification questions per query
- If no unresolved ambiguity/preference conflicts, skip straight to synthesis
- No model call — the dispute objects already contain everything needed

**Self-preference bias:** The synthesis model might favor analysis from the same architecture. No V1 mitigation — accepted risk (see V1 Limitations). Model assignment strategy partially mitigates via framing diversity.

**Context compaction:** Before synthesis, compress prior-stage artifacts into a distilled input. Original query at start, key disputes and evidence at end. Char count heuristic: compress when input exceeds ~80K chars (~20K tokens). LiteLLM `token_counter` is a trivial upgrade if synthesis quality is poor.

**Synthesis adapts based on dispute resolution:**
- Empirical disputes resolved → converge on best-supported recommendation
- Interpretive disputes → present strongest interpretation, note alternatives
- Preference-weighted → surface 2-3 viable paths with explicit "if X then Y" conditions
- Remaining ambiguity → flag honestly as unresolved

Always: lead with strongest recommendation, present alternatives as secondary.

**Output:** Final report per the Output Contract (11 components, 3 tiers) + trace file.

## Model Roles

| Stage | Model | Why |
|---|---|---|
| 1. Decomposition | Gemini 3.1 Pro | Best cost-performance; saves budget for high-value stages |
| 2. Evidence search | Orchestration layer (Tavily/Exa) | Not a model task |
| 3. Independent analysis (landscape) | GPT-5.4 | Strongest holistic reasoning; anti-sycophancy training |
| 3. Independent analysis (critical) | Claude Opus 4.6 | Constitutional AI; reflective reasoning |
| 3. Independent analysis (verification) | Gemini 3.1 Pro | MoE architecture; different reasoning patterns |
| 4. Claim extraction + dispute localization | GPT-5.4 | Best structured output (constrained decoding) |
| 5. Arbitration critique | Claude Opus 4.6 | Lowest prompt injection rate; reflective; single-turn |
| 5. Targeted search | Orchestration layer (Tavily/Exa) | Dispute-driven queries |
| 6. Synthesis | TBD | Validate in testing. Must not dominate earlier stages. |

**Note:** Role assignments are configuration, not architecture. Reassign as models update.

**Gemini 3.1 Pro stability warning:** Still in preview as of March 2026 (503 errors, latencies up to 104s, GA estimated April-May 2026). Fallback: GPT-5.4 mini or Claude Sonnet 4.6 for Gemini's roles (Stage 1 decomposition, Stage 3 verification frame). Monitor status before build.

## Disagreement Classification Framework

Every disagreement between models is classified into one of five types.

| Type | What it is | How to resolve |
|---|---|---|
| Empirical | Dispute about facts or what sources show | More evidence / targeted search (Stage 5) |
| Interpretive | Dispute about what evidence means | Arbitration critique (Stage 5) |
| Preference-weighted | Dispute about optimization criteria | User context or "if X then Y" (Stage 6 interrupt) |
| Spec ambiguity | Question not constrained enough | Structured user steering (Stage 6 interrupt) |
| Other | Edge case not fitting above types | Routes to user input (Stage 6a). Uses description field for context. |

## Output Contract

A successful run returns **11 components in three tiers** plus a separate trace file.

### Tier A: Executive (what the user reads first)
1. **Executive recommendation** (1–3 paragraphs) — Bottom-line answer as a recommendation, not a survey.
2. **Conditions of validity** — What would flip the recommendation. Stated upfront.
3. **Decision-relevant tradeoffs** — "If you optimize for X → A. If you optimize for Y → B."

### Tier B: Analytical (what the user reads for understanding)
4. **Disagreement map** — Types of disagreements found, action taken on each.
5. **Preserved alternatives** — Only alternatives that survived critique, with conditions for preference.
6. **Key assumptions** — For each: if wrong, what changes?
7. **Confidence assessment** — Per major claim: high / medium / low.
8. **Process summary** — Per-stage bullets (4-6 each) covering: what the stage was trying to do, what key conflicts/tradeoffs it found, what it decided, and what it produced. Scannable in 30 seconds. Compiled from stage summary objects emitted during the pipeline run.

### Tier C: Evidentiary (what the user reads for verification)
9. **Claim ledger excerpt** — Decision-critical claims with final status and resolution path.
10. **Evidence trail** — Sources, quality scores, conflicts and how resolved.
11. **Evidence gaps** — What the system looked for but couldn't find.

### Trace File (separate from report)
Full detailed log of every stage's artifacts, raw claim ledger, dispute queue, evidence sources, and resolution paths. Available for debugging and deep inspection. Not part of the main report — generated alongside it.

### Output anti-patterns (must NOT contain)

**Validated at runtime (V1):**
- Claims not traceable to evidence — grounding check (regex validation)
- Alternatives eliminated by critique but included for balance — zombie check

**Eval criteria (Phase 5, not V1 runtime):**
- Apparent consensus masking unresolved disagreements (false consensus)
- Hedging without specifics ("it depends" without saying on what)
- More than 2 levels of conditional nesting
- Re-opening disputes that were resolved in Stage 5
- Replacing claim-level uncertainty with polished prose — unresolved stays unresolved

## Source Strategy (Locked)

**Search APIs:** Tavily + Exa (routed by query type)
- Tavily: keyword search, news/current events, content extraction from known URLs
- Exa: semantic/conceptual queries, academic papers (`category: "research paper"`)
- Both in parallel for high-priority sub-questions, merged and deduped by normalized URL
- Domain filtering via API parameters: default `exclude_domains` for aggregators, optional `include_domains` per topic

### Tier 1: Breadth Scouting (Stage 2)
- 3-4 query variants per sub-question via lightweight model call (keyword, practitioner, contrarian, + Exa semantic)
- Practitioner-signal patterns: `"we found that"`, `"lessons learned"`, `site:github.com`
- Avoid: "best practices for X," "ultimate guide to Y"
- URL deduplication before scoring (normalize URLs, strip tracking params)
- Source quality scoring: URL lookup table (authority) blended with freshness decay + staleness flags → final `quality_score`. Authority floor of 0.4 for seminal sources. Unknown sources default to 0.5 (not penalized).
- Evidence sufficiency: min 2 independent sources per sub-question (checked after dedup)
- Basic compression: extract findings, preserve key quotes

### Tier 2: Dispute-Driven Targeting (Stage 5)
- Fires only on disputes that are both empirical/interpretive AND decision-critical
- Neutral verification queries (AVeriTeC-style): neutral question + supporting search for weaker position + authoritative source query
- Tavily `search_depth="advanced"` with `chunks_per_source=3` for targeted content
- Budget: max 3 queries per disputed claim, max 2 rounds total
- Recency-sensitive claims require official source OR 2 independent timestamped sources

## Unit of Analysis (Locked)

**Claim–assumption–scenario tuples**, with claims as the operational core.

1. **Atomic claim** — Specific, falsifiable. Primary compute surface.
2. **Assumption** — First-class citizen, subject to same evidence mapping as claims.
3. **Scenario** — Only where it would flip the recommendation. Kept minimal.

**Alternatives** tracked as bundles of claims + assumptions + scenarios.

## Unit of Disagreement (Locked)

**Typed claim-level conflict, post-filtered by decision-criticality.**

Five-type taxonomy with deterministic routing. Decision-criticality gate prevents wasting resources on peripheral disputes. Dispute records are derived work queues materialized from the claim graph — not a parallel data store.

## Cross-Cutting Requirements

**`reasoning` field:** All model-output schemas must include a `reasoning` field — how the model got from input to output. Reasoning IS the product in this pipeline. Applies to: StageSummary, AnalysisObject, SynthesisReport. (ArbitrationAssessment already has it.)

**Fallback logic:** Three must-succeed stages with fallback chains:
- Stage 1: Gemini → GPT
- Stage 4: GPT → Claude
- Stage 6b: primary → alternate
- Stage 3 requires ≥2 of 3 models to succeed. Everything else skips gracefully.

**Partial trace on abort:** If the pipeline fails mid-run, emit a trace file with everything completed so far. Never lose the work.

**Anonymization:** Enforced everywhere models see each other's output. String replacement, no model identities leak.

## V1 Limitations & Accepted Tradeoffs

*Deliberate tradeoffs for a personal CLI tool. Would need addressing for a product.*

- **No decomposition validation (Stage 1v deferred).** If Stage 1 subtly frames the question in a biased way, that bias flows through the entire pipeline. The self-check prompt and terminal review catch obvious problems, but not subtle ones.
- **No echo detection (deferred).** Multiple sources citing the same upstream study look like independent confirmation. The pipeline could report "3 sources confirm X" when really it's 1 source cited 3 times. Search APIs mostly return diverse results, so this is rare in practice.
- **Lighter output validation.** Only 2 of 7 anti-patterns are automatically checked at runtime (grounding + zombie). The other 5 are eval criteria, not runtime guards.
- **Counter-argument specificity downgraded.** Prompt asks for specific, evidence-linked counter-arguments, but no automated check enforces it (~70-80% effective with frontier models).
- **No self-preference bias guard.** Synthesis model might favor analysis from the same architecture. Accepted risk — model assignment partially mitigates via framing diversity.
- **Approximate context management.** Char count heuristic instead of precise token counting. Good enough for V1.
- **Error amplification risk in parallel architecture.** Research on independent multi-agent systems shows error amplification up to 17.2x without structured cross-referencing (Google Research, 2025). Stages 4-6 are the mitigation — they are not optional stages, they are the error-correction mechanism that makes the parallel architecture net-positive. If Stage 4 extraction quality degrades, the pipeline can produce worse results than a single model. This is why Stage 4 is labeled quality bottleneck #2.
- **Interaction effects:** No Stage 1v + no echo detection = biased decomposition AND inflated evidence can both slip through unchecked. Lighter synthesis checks compound this. Three Stage 3 models all analyze the same potentially biased evidence.
