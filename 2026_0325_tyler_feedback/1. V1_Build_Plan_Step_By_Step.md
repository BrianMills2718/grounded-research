# V1 Build Plan — Step by Step

> The complete stage-by-stage specification for V1. Each stage lists exactly what it does, what model runs it, what it produces, and any constraints. Build this.

---

## Stage 1 — Intake & Decomposition

**Model:** Gemini 3.1 Pro
**Fallback:** GPT-5.4 mini or Claude Sonnet 4.6 (Gemini is still in preview — if unstable, swap at config level)

**What it does:** Takes the user's raw question and breaks it into structured components the rest of the pipeline operates on.

**Steps:**
1. Restate the query — show the restated question in terminal so the user can re-run if it's wrong
2. Break into empirical sub-questions (what can be researched)
3. Map optimization axes / tradeoffs (feeds Stage 6 tradeoff output)
4. Generate research plan: what to verify, what sources matter, what evidence would change the recommendation
5. Generate falsification targets: what would prove the recommendation wrong (N/A is valid for non-falsifiable topics)
6. Prompt self-check: "Does this decomposition address the FULL question? Are sub-questions neutral, not leading?"
7. Emit stage summary with `reasoning` field

**Output artifacts:**
- Decomposition object (sub-questions, optimization axes, research plan, falsification targets)
- Stage summary (goal, key findings, decisions, outcome, reasoning)

**Note:** No decomposition validation stage in V1. The self-check prompt instruction and user terminal review are the mitigations.

---

## Stage 2 — Broad Retrieval & Evidence Normalization

**Tools:** Tavily (primary, structured JSON) + Exa (secondary, semantic/neural)

**What it does:** For each sub-question from Stage 1, searches the web for real evidence and normalizes it into a structured evidence package.

**Steps:**
1. Generate 3-5 query variants per sub-question (formal/colloquial, academic/practitioner, broad/narrow) — string templates, not a model call
2. Execute searches — hard cap: 4 queries per sub-question (named constant)
3. Apply source quality scoring — URL lookup table (1.0 official docs → 0.3 generic blog). Unknown sources default to 0.5 (not penalized). Note: Tavily's `score` field is relevance, not credibility.
4. Extract atomic findings from each source — this is quality bottleneck #1, no shortcuts
5. Basic compression: keep findings and preserve key quotes
6. Check evidence sufficiency: minimum 2 independent sources per sub-question
7. Emit stage summary with `reasoning` field

**Practitioner-signal patterns:** `"we found that"`, `"lessons learned"`, `site:github.com`
**Avoid:** "best practices for X," "ultimate guide to Y"

**Output artifacts:**
- Evidence package (normalized sources with quality scores, per sub-question)
- Stage summary (goal, sources found, quality assessment, gaps identified, reasoning)

## Stage 3 — Independent Candidate Generation

**Models (parallel, truly independent):**
- GPT-5.4 — landscape frame (step-back abstraction)
- Claude Opus 4.6 — critical frame (structured decomposition)
- Gemini 3.1 Pro — verification frame (verification-first)

**Fallback:** If Gemini is unstable, use GPT-5.4 mini or Claude Sonnet 4.6. Stage requires ≥2 of 3 models to succeed.

**What it does:** Each model independently analyzes the evidence package using a different reasoning frame. No model sees another's output.

**Each model must produce:**
1. Bottom-line recommendation (prevents hedging)
2. Specific, evidence-linked claims
3. Explicit assumptions
4. Evidence references (linked to Stage 2 sources)
5. Counter-argument against their own recommendation (required non-empty string — prompt asks for specifics but no automated validation)
6. What would falsify their recommendation
7. Confidence level (HIGH/MEDIUM/LOW — telemetry only, not arbitration input)
8. `reasoning` field

**Anonymization:** Outputs are labeled A/B/C via string replacement. No model identity leaks downstream.
**Anti-conformity:** Models may only change position when citing new evidence, a corrected assumption, or a resolved contradiction. Prompt never asks "who is more convincing."

**Output artifacts:**
- Three independent analysis objects (structured JSON, anonymized A/B/C)
- Stage summary per model (goal, key claims, reasoning approach, confidence level, reasoning)

## Stage 4 — Claim Extraction & Dispute Localization

**Model:** GPT-5.4 (best structured output via constrained decoding)
**Fallback:** Claude Opus 4.6

**What it does:** Decomposes the three independent analyses into atomic claims, identifies where they conflict, classifies each conflict by type, and filters for decision-criticality. This is quality bottleneck #2.

**Steps:**

**4a. Claim extraction:**
1. Decompose each model's analysis into atomic claims using Claimify approach (sentence tokenization → context creation → LLM extraction → disambiguation)
2. Extract assumptions as first-class claims
3. Deduplicate claims across the three analyses (LLM-based, not programmatic)
4. Assign global IDs and evidence labels to all claims (evidence label hierarchy: vendor-documented > empirically observed > model self-characterization > speculative inference)

**4b. Dispute localization:**
5. Identify where the three analyses conflict
6. Classify each conflict:
   - **Empirical** — dispute about facts → resolve with more evidence (Stage 5)
   - **Interpretive** — dispute about what evidence means → arbitration critique (Stage 5)
   - **Preference-weighted** — dispute about optimization criteria → user context needed (Stage 6a)
   - **Spec ambiguity** — question not constrained enough → user steering (Stage 6a)
   - **Other** — edge case not fitting above types, with free-text description field

**4c. Decision-criticality gate:**
7. Filter: does this disagreement affect which alternative is recommended? If not, log but don't route to resolution.

**4d. Resolution routing:**
8. Route disputes deterministically based on type (empirical/interpretive → Stage 5, preference/ambiguity → Stage 6a)

9. Emit stage summary with `reasoning` field

**Output artifacts (all materialized, not hidden in prompt state):**
- Claim ledger: all extracted claims with source model (anonymized), evidence links, status
- Assumption set: all extracted assumptions, which claims depend on them
- Dispute queue: classified disagreements with type, criticality flag, resolution routing
- Stage summary (claims extracted count, disputes found, type breakdown, what got filtered, reasoning)

## Stage 5 — Targeted Verification & Arbitration

**Tools:** Tavily/Exa for evidence search
**Model for arbitration:** Claude Opus 4.6 (single-turn, schema-driven critique)

**What it does:** Only fires on unresolved disputes from Stage 4 that are both empirical/interpretive AND decision-critical. Low-conflict queries skip this stage entirely.

**Steps:**
1. For each routed dispute, generate counterfactual queries aimed at refuting the leading claim
2. Search for both supporting AND disconfirming evidence
3. Arbitration: update claim status based on new evidence (supported / contradicted / insufficient)
4. Update claim ledger and dispute queue with resolution status
5. Emit stage summary with `reasoning` field

**Budget:** Max 3 queries per disputed claim. Max 2 rounds total.

**Recency-sensitive claims:** Require official source OR 2 independent timestamped sources.

**Counterfactual query patterns:** `[topic] limitations`, `[claim] contradicted by`

**Output artifacts:**
- Updated claim ledger + dispute queue with resolution status
- Stage summary (disputes investigated, evidence found, what changed, what remains unresolved, reasoning)

## Stage 6a — User-Steering Interrupt

**What it does:** Before synthesis, checks the dispute queue for unresolved spec ambiguity or preference-weighted conflicts. If found, asks the user for input. If none found, skips straight to Stage 6b.

**Steps:**
1. Python filter on dispute queue: any unresolved preference-weighted or spec-ambiguity disputes?
2. If yes: print dispute info to terminal (what's ambiguous, the options, why it matters)
3. User provides free-text input or presses Enter for default
4. Max 2 clarification questions per query

**No model call.** No schemas. Just terminal I/O using data already in the dispute objects.

---

## Stage 6b — Synthesis & Final Report

**Model:** TBD (validate Claude Opus 4.6 vs GPT-5.4 in testing — must not be the model that dominated earlier stages)
**Fallback:** If primary fails, try alternate model.

**What it does:** Compresses prior-stage artifacts, synthesizes the final report adapting to dispute types, and runs output validation.

**Steps:**

**Context compaction:**
1. Compress prior-stage artifacts when input exceeds ~80K characters (~20K tokens). Original query at start, key disputes and evidence at end.

**Synthesis (adapts based on dispute resolution):**
2. Empirical disputes resolved → converge on best-supported recommendation
3. Interpretive disputes → present strongest interpretation, note alternatives
4. Preference-weighted → surface 2-3 viable paths with explicit "if X then Y" conditions
5. Remaining ambiguity → flag honestly as unresolved
6. Always: lead with strongest recommendation, present alternatives as secondary

**Output validation (2 automated checks):**
7. Grounding check (regex): does the recommendation cite claims from the claim ledger?
8. Zombie check: are any alternatives that were eliminated by critique still included?

9. Emit stage summary and `reasoning` field

**Output: Final report (11 components, 3 tiers) + trace file**

### Tier A: Executive (what the user reads first)
1. **Executive recommendation** (1-3 paragraphs) — bottom-line answer as a recommendation, not a survey
2. **Conditions of validity** — what would flip the recommendation
3. **Decision-relevant tradeoffs** — "If you optimize for X → A. If you optimize for Y → B."

### Tier B: Analytical (what the user reads for understanding)
4. **Disagreement map** — types of disagreements found, action taken on each
5. **Preserved alternatives** — only alternatives that survived critique, with conditions for preference
6. **Key assumptions** — for each: if wrong, what changes?
7. **Confidence assessment** — per major claim: HIGH / MEDIUM / LOW
8. **Process summary** — per-stage bullets (4-6 each) compiled from stage summaries. Scannable in 30 seconds.

### Tier C: Evidentiary (what the user reads for verification)
9. **Claim ledger excerpt** — decision-critical claims with final status and resolution path
10. **Evidence trail** — sources, quality scores, conflicts and how resolved
11. **Evidence gaps** — what the system looked for but couldn't find

### Trace File (separate from report)
Full log of every stage's artifacts, raw claim ledger, dispute queue, evidence sources, resolution paths. For debugging and deep inspection.

## Cross-Cutting Requirements

### Reasoning field
All model-output schemas must include a `reasoning` field. Applies to: StageSummary, AnalysisObject, SynthesisReport, ArbitrationAssessment.

### Fallback logic
Three must-succeed stages:
- Stage 1: Gemini → GPT-5.4 mini
- Stage 4: GPT-5.4 → Claude Opus 4.6
- Stage 6b: primary → alternate

Everything else skips gracefully. Stage 3 requires ≥2 of 3 models to succeed.

### Partial trace on abort
If the pipeline fails at any stage, emit a trace file with everything completed so far.

### Anonymization
String replacement of model identifiers before any cross-model comparison. No model knows which model produced which output.

### Anti-conformity
A model may only change position when citing new evidence, a corrected assumption, or a resolved contradiction. Critique prompts ask for flaws and missing evidence, never "who is more convincing."

## Known Limitations

These are accepted tradeoffs for a personal CLI tool:

- **No decomposition validation** — biased framing in Stage 1 passes through unchecked (mitigated by prompt self-check + user terminal review)
- **No echo detection** — multiple sources citing the same upstream study look like independent sources
- **Lighter output validation** — 2 of 7 anti-patterns checked at runtime (grounding + zombie). The other 5 are evaluation criteria, not runtime checks.
- **Counter-argument quality is prompt-only** — no automated check that counter-arguments are specific and evidence-linked
- **No self-preference bias guard** — synthesis model may favor analysis from same architecture
- **Approximate context sizing** — char count heuristic instead of token counting
- **Compounding risk** — no decomposition validation + no echo detection means biased framing AND inflated evidence can both pass through unchecked
